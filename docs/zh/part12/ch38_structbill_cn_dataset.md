# 第38章：StructBill-CN 票据文档理解数据工程

## 源文件

- 原始材料：`StructBill-CN_Dataset.docx`

## 章节定位

本章用于把合作老师的 StructBill-CN 工作整理为“高风险中文票据与医疗费用文档的数据工程案例”。它不是单纯介绍一个数据集，而是要说明：在视觉文档理解任务中，如何从真实票据图像、schema、层级 JSON、表格字段和逻辑约束出发，构建可训练、可评测、可复查的数据资产。

本章承接第三篇的文档理解与 OCR、多模态解析，第七篇的 RAG/文档证据组织，第八篇的数据版本和实验追踪，第九篇的数据资产与契约治理，以及第十一篇的合规边界。向后则为第十三篇的 VLM 数据配方和第十四篇的多模态 RAG、隐私流水线项目提供一个中文垂直文档数据案例。

## 写作目标

- 讲清 StructBill-CN 的任务边界：schema-based unified information extraction、visual document understanding、logical consistency。
- 讲清样本结构：图像、预定义 schema、全局 key-value 字段、嵌套 line-item table、逻辑约束。
- 讲清构建流程：数据来源、清洗、schema 设计、标注规范、逻辑校验、质检和数据切分。
- 讲清评测方式：字段级准确率、结构一致性、表格行列对齐、算术一致性、schema 约束违反率。
- 讲清工程价值：为什么中文票据/医疗费用文档适合作为高风险结构化抽取 benchmark。

## 素材使用要求

- 必须以 docx 中给出的数据规模、schema 类型、样例 JSON 和 SRPO 设计关系为基础，不要凭空扩展数据事实。
- 如果学生补充论文、代码或仓库信息，必须标注来源，并区分“原始材料已有信息”和“补充调研信息”。
- 不要把本章写成模型论文综述，重点放在数据工程对象、数据流、质量控制和可复现评测。

## 建议结构

1. 问题场景：票据和医疗费用文档为什么难做结构化抽取。
2. 数据集概览：规模、来源、业务 schema、图像类型和任务定义。
3. 样本 schema：key-value、line-item table、层级 JSON 与逻辑约束。
4. 构建流水线：采集、去噪、标注、schema 对齐、逻辑校验、版本切分。
5. 评测协议：字段、结构、表格、算术一致性和错误归因。
6. 工程复盘：适合训练什么模型，适合评估什么能力，不适合滥用到哪些任务。
7. 与前后章节回链：连接 Ch09、Ch21、Ch25、Ch27-Ch30、Ch47 和 P05/P09。
